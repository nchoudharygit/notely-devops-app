# Notely — Application Specification

**Version:** 1.0  
**Type:** Multi-tier web application  
**Purpose:** A simple Notes/TODO application used as a learning vehicle for cloud infrastructure deployment

---

## 1. Overview

Notely is a single-user or multi-user notes app. Users can sign up, log in, and manage personal notes. Each note has a title, a body, optional tags, and can have one or more file attachments. The app exposes a REST API consumed by a web frontend.

---

## 2. User Stories

### Authentication

- As a user, I can register with an email and password.
- As a user, I can log in and receive a session token.
- As a user, I can log out, which invalidates my session.

### Notes

- As a user, I can create a new note with a title and body.
- As a user, I can view a list of all my notes (paginated).
- As a user, I can view a single note by ID.
- As a user, I can edit the title and body of a note.
- As a user, I can delete a note.
- As a user, I can search my notes by keyword (title or body).

### Tags

- As a user, I can create a tag.
- As a user, I can attach one or more tags to a note.
- As a user, I can filter my notes by tag.
- As a user, I can rename or delete a tag.

### Attachments

- As a user, I can upload a file (image or PDF) to a specific note.
- As a user, I can view a list of attachments on a note.
- As a user, I can download an attachment.
- As a user, I can delete an attachment from a note.

## 3. System Components

### 3.1 Web Server

Serves the frontend application to the browser. Handles no business logic directly.

- Delivers HTML, CSS, and JavaScript assets.
- All API communication goes from the browser to the API server — the web server does not proxy API calls.
- Serves a single `/health` endpoint that returns `200 OK` for load balancer health checks.

### 3.2 API Server

The core backend service. Stateless by design — all state lives in the database, cache, or object storage.

- Exposes a JSON REST API (see Section 5).
- Handles authentication via session tokens stored in Redis.
- Delegates file storage to the object storage service.
- Reads and writes all persistent data to the database.
- Applies rate limiting per user using counters in Redis.

### 3.3 Object Storage

Stores binary files — frontend build artifacts and user-uploaded note attachments.

- **Static assets bucket:** Contains the compiled frontend (HTML, JS, CSS, images). Writable only during deployment; readable by the web server.
- **Attachments bucket:** Contains files uploaded by users. Private; accessed only through pre-signed URLs generated by the API server. Files are never publicly accessible directly.

### 3.4 Database (PostgreSQL)

The system of record for all structured data.

- Stores users, notes, tags, and attachment metadata.
- The API server is the only component that connects to the database.
- Credentials are not hardcoded; they are read from a secrets store at startup.

### 3.5 Cache (Redis)

Used for three purposes:

1. **Session store:** After login, the API stores `session:{token} → user_id` with a 24-hour TTL. Every authenticated request reads from Redis to validate the token.
2. **Note cache:** Recently fetched notes are cached as `note:{id} → JSON` with a 5-minute TTL. Cache entries are invalidated immediately on update or delete.
3. **Rate limiting:** Per-user per-minute request counters are stored as `ratelimit:{user_id}:{epoch_minute}` with a 60-second TTL. If the counter exceeds 60, the API returns `429 Too Many Requests`.

## 4. Data Model

### `users`


| Column        | Type         | Notes           |
| ------------- | ------------ | --------------- |
| id            | UUID         | Primary key     |
| email         | VARCHAR(255) | Unique, indexed |
| password_hash | TEXT         | bcrypt          |
| created_at    | TIMESTAMPTZ  |                 |


### `notes`


| Column     | Type         | Notes                  |
| ---------- | ------------ | ---------------------- |
| id         | UUID         | Primary key            |
| user_id    | UUID         | FK → users.id, indexed |
| title      | VARCHAR(255) |                        |
| body       | TEXT         |                        |
| created_at | TIMESTAMPTZ  |                        |
| updated_at | TIMESTAMPTZ  | Updated on every edit  |


### `tags`


| Column  | Type         | Notes           |
| ------- | ------------ | --------------- |
| id      | UUID         | Primary key     |
| user_id | UUID         | FK → users.id   |
| name    | VARCHAR(100) | Unique per user |


### `note_tags`


| Column            | Type | Notes                 |
| ----------------- | ---- | --------------------- |
| note_id           | UUID | FK → notes.id         |
| tag_id            | UUID | FK → tags.id          |
| (note_id, tag_id) |      | Composite primary key |


### `attachments`


| Column       | Type         | Notes                                  |
| ------------ | ------------ | -------------------------------------- |
| id           | UUID         | Primary key                            |
| note_id      | UUID         | FK → notes.id, indexed                 |
| storage_key  | TEXT         | The key/path in the attachments bucket |
| filename     | VARCHAR(255) | Original filename as uploaded          |
| content_type | VARCHAR(100) | MIME type (e.g. `image/jpeg`)          |
| size_bytes   | BIGINT       |                                        |
| uploaded_at  | TIMESTAMPTZ  |                                        |

## 5. API Specification

**Base URL:** `/api/v1`  
**Auth:** All endpoints except `POST /auth/register` and `POST /auth/login` require a header:  
`Authorization: Bearer <session_token>`

### Authentication


| Method | Path             | Description                          | Request Body          | Response                    |
| ------ | ---------------- | ------------------------------------ | --------------------- | --------------------------- |
| POST   | `/auth/register` | Create a new user                    | `{ email, password }` | `201 { id, email }`         |
| POST   | `/auth/login`    | Login, receive a session token       | `{ email, password }` | `200 { token, expires_at }` |
| POST   | `/auth/logout`   | Invalidate the current session token | —                     | `204`                       |


### Notes


| Method | Path         | Description                         | Query / Body                             | Response                                  |
| ------ | ------------ | ----------------------------------- | ---------------------------------------- | ----------------------------------------- |
| GET    | `/notes`     | List all notes for the current user | `?page=1&limit=20&tag=tagname&q=keyword` | `200 { data: [...], total, page, limit }` |
| POST   | `/notes`     | Create a note                       | `{ title, body, tag_ids?: [] }`          | `201 { note }`                            |
| GET    | `/notes/:id` | Get a single note                   | —                                        | `200 { note }`                            |
| PUT    | `/notes/:id` | Replace a note entirely             | `{ title, body, tag_ids?: [] }`          | `200 { note }`                            |
| PATCH  | `/notes/:id` | Partial update                      | `{ title?, body?, tag_ids?: [] }`        | `200 { note }`                            |
| DELETE | `/notes/:id` | Delete a note and its attachments   | —                                        | `204`                                     |


### Tags


| Method | Path        | Description                            | Request Body | Response              |
| ------ | ----------- | -------------------------------------- | ------------ | --------------------- |
| GET    | `/tags`     | List all tags for the current user     | —            | `200 { data: [...] }` |
| POST   | `/tags`     | Create a tag                           | `{ name }`   | `201 { tag }`         |
| PUT    | `/tags/:id` | Rename a tag                           | `{ name }`   | `200 { tag }`         |
| DELETE | `/tags/:id` | Delete a tag (detaches from all notes) | —            | `204`                 |


### Attachments


| Method | Path                                            | Description                     | Request Body                | Response                  |
| ------ | ----------------------------------------------- | ------------------------------- | --------------------------- | ------------------------- |
| GET    | `/notes/:id/attachments`                        | List attachments on a note      | —                           | `200 { data: [...] }`     |
| POST   | `/notes/:id/attachments`                        | Upload a file                   | `multipart/form-data: file` | `201 { attachment }`      |
| GET    | `/notes/:id/attachments/:attachmentId/download` | Get a time-limited download URL | —                           | `200 { url, expires_at }` |
| DELETE | `/notes/:id/attachments/:attachmentId`          | Delete an attachment            | —                           | `204`                     |

## 6. Request/Response Flow

### Login and fetch a note

```
Browser
  │
  │ POST /api/v1/auth/login
  ▼
API Server
  │── Reads user row from PostgreSQL (verify bcrypt hash)
  │── Generates random token
  │── SET session:{token} user_id  (Redis, TTL 24h)
  └── Returns { token }

Browser
  │
  │ GET /api/v1/notes/abc123  (Authorization: Bearer <token>)
  ▼
API Server
  │── GET session:{token}  → user_id  (Redis)
  │── GET note:abc123  (Redis)
  │     hit?  → return cached JSON
  │     miss? → SELECT * FROM notes WHERE id=abc123
  │             SET note:abc123 <json>  (Redis, TTL 5m)
  └── Returns { note }
```

### Upload an attachment

```
Browser
  │
  │ POST /api/v1/notes/abc123/attachments  (multipart body)
  ▼
API Server
  │── Authenticate via Redis session
  │── Stream file to Object Storage (attachments bucket)
  │── INSERT INTO attachments (note_id, storage_key, filename, ...)
  └── Returns { attachment }
```

## 7. Non-Functional Requirements


| Concern            | Requirement                                                                                |
| ------------------ | ------------------------------------------------------------------------------------------ |
| Authentication     | Session-based (token in Authorization header); tokens expire in 24h                        |
| Rate limiting      | 60 requests/minute per user; exceeding returns `429`                                       |
| File size limit    | Max 10 MB per attachment                                                                   |
| Allowed file types | `image/jpeg`, `image/png`, `image/gif`, `application/pdf`                                  |
| Pagination         | All list endpoints support `page` and `limit`; default limit is 20, max is 100             |
| Search             | Keyword search on notes uses a PostgreSQL `ILIKE` query (full-text search is out of scope) |
| Passwords          | Minimum 8 characters; stored as bcrypt hash (cost factor 12)                               |
| HTTPS              | All traffic encrypted in transit; API server assumes TLS termination is handled upstream   |

## 8. Error Response Format

All errors follow a consistent envelope:

```json
{
  "error": {
    "code": "NOTE_NOT_FOUND",
    "message": "No note with id 'abc123' exists for this user.",
    "status": 404
  }
}
```

Standard error codes:


| HTTP Status | Code                     | Meaning                                          |
| ----------- | ------------------------ | ------------------------------------------------ |
| 400         | `VALIDATION_ERROR`       | Missing or invalid request field                 |
| 401         | `UNAUTHORIZED`           | Missing or invalid session token                 |
| 403         | `FORBIDDEN`              | Authenticated but not the owner of this resource |
| 404         | `NOT_FOUND`              | Resource does not exist                          |
| 409         | `CONFLICT`               | Duplicate (e.g. email already registered)        |
| 413         | `FILE_TOO_LARGE`         | Attachment exceeds 10 MB                         |
| 415         | `UNSUPPORTED_MEDIA_TYPE` | File type not allowed                            |
| 429         | `RATE_LIMITED`           | Exceeded 60 req/min                              |
| 500         | `INTERNAL_ERROR`         | Unexpected server error                          |
